# L'injection de dépendances

----

## L'IoC (Inversion de Contrôle)

L'Inversion de Contrôle est un principe d'architecture où le flux d'exécution du programme n'est plus dirigé par le code, mais par le framework sous-jacent.

Dans le contexte de Spring (qui est un conteneur IoC) :

- Au lieu de faire un `new MonService()`, le framework va créer cet objet au démarrage, on parle d'Injection de Dépendances
- L'Inversion de Contrôle retire aux classes la responsabilité de gérer la création de leurs propres dépendances

----

## L'Inversion des Dépendances (Le "D" de SOLID)

L'Inversion des Dépendances est un principe de conception (le dernier des principes SOLID) qui stipule deux règles fondamentales :

- Les modules de haut niveau ne doivent pas dépendre des modules de bas niveau. Les deux doivent dépendre d'abstractions (des interfaces).
- Les abstractions ne doivent pas dépendre des détails d'implémentation. Ce sont les détails qui doivent dépendre des abstractions.

**L'idée principale** : Le code ne doit pas être couplé à une implémentation spécifique, mais à un "contrat".

- Exemple sans inversion (Couplage fort) : Un service `GestionPaiement` dépend directement d'une classe `PaiementParCarte`. Si demain on veut passer à `PayPal`, il faut réécrire le service.
- Exemple avec inversion (Couplage faible) : Le service `GestionPaiement` dépend d'une interface `MethodePaiement`. Les classes `PaiementParCarte` et `PaiementPayPal` implémentent cette interface. Le service se moque de savoir quelle implémentation est utilisée, il sait juste comment interagir avec le contrat.

----

## Le Couplage Fort : L'instanciation manuelle

Avant l'Inversion de Contrôle (IoC) ou dans du code sans framework, le composant gérait lui-même la création de ses dépendances.

```java
public class MonController {
    private MonService monService;

    public MonController() {
        this.monService = new MonService(); // Création manuelle
    }
}
```

- Avantage : très facile à lire et à faire
- Inconvéniant : couplage fort, impossible de remplacer `MonService` par un faux objet pour les tests, le code viole le principe SOLID de l'inversion de dépendances
- **Verdict** : à proscrire

----

## La "Magie" Spring : l'injection par champ

Avec l'arrivée de Spring, on a confié la création des objets au framework, qui fait de l'injection par réflexion.

```java
public class MonController {
    @Autowired
    private MonService monService;
}
```

- Avantage : code extrêmement concis
- Inconvéniant : rend les tests unitaires plus difficiles et les variables ne peuvent pas être `final`
- **Verdict** : toléré sur les POC, mais considéré comme une mauvaise pratique aujourd'hui

----

## Le Retour aux Sources : L'injection par Constructeur

L'ingénierie logicielle a réalisé que le code "pur Java" était plus sain. On passe la dépendance à la création de l'objet.

```java
public class MonController {
    private final MonService monService;

    // @Autowired (Facultatif si un seul constructeur)
    public MonController(MonService monService) {
        this.monService = monService;
    }
}
```

- Avantage : code robuste (le champ est final), rend la classe testable très facilement avec un simple `new MonController(mock)`, et empêche les dépendances circulaires
- Inconvéniant : verbeux
- **Verdict** : la méthode officielle recommandée par Spring pour des applications fiables et testables

----

## L'Élégance Moderne : L'injection avec Lombok

Pour garder tous les avantages de l'architecture du constructeur tout en effaçant sa verbosité, l'écosystème Java a adopté la librairie Lombok. L'annotation `@RequiredArgsConstructor` génère le constructeur à la compilation pour tous les champs marqués comme **final**.

```java
@RequiredArgsConstructor
public class MonController {
    private final MonService monService;
    // Le constructeur est généré en coulisses !
}
```

- Avantage : le meilleur des 2 mondes. On garde l'immutabilité (final), la testabilité et les standards Spring, mais avec un code aussi concis que l'injection par champ historique
- Inconvéniant : Nécessite d'ajouter la dépendance Lombok au projet
- **Verdict** : Le standard *de facto* dans la quasi-totalité des projets Spring Boot modernes.

----

## L'intérêt des variables final

- **L'immutabilité** : une fois que la variable final a été assignée dans le constructeur, sa référence ne peut plus jamais changer durant toute la durée de vie de l'objet. L'état du composant est donc verrouillé et prévisible.
- **L'anti-bug** en empêchant la réassignation, tu élimines le risque qu'un autre développeur ne modifie par erreur la dépendance avec un `this.monService = null;` au milieu d'une méthode.
- **Le Thread-Safety** : le controller va recevoir des dizaines de requêtes HTTP en même temps, exécutées sur des threads différents. Si le contrôleur possède un état mutable (des variables non final qui peuvent changer), ces requêtes risquent de se marcher sur les pieds et de causer des comportements imprévisibles. Un composant immutable est naturellement à l'abri de ces problèmes de concurrence.
